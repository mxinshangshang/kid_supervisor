## 2. v4.2 关键问题与处理

### 2.1 通知照片颜色反转

问题：
1. 飞书通知中的照片颜色是反转的（RGB vs BGR）
2. 原因是保存照片时进行了重复的颜色空间转换

处理：
1. 统一 `save_photo()` 函数，直接保存接收到的 RGB 帧
2. 移除多余的 `cv2.cvtColor(frame_rgb, cv2.COLOR_RGB2BGR)` 转换

### 2.2 进入/退出学习模式没有拍照

问题：
1. 只有告警时才拍照
2. 学习开始和结束没有照片记录

处理：
1. 在检测到人进入时（学习开始）拍照
2. 在检测到人离开时（学习结束）拍照
3. 照片随通知一起发送到飞书

### 2.3 告警消息显示所有问题而非最严重的

问题：
1. 告警会列出所有检测到的姿势问题
2. 不够直观，用户不清楚哪个是最严重的

处理：
1. 在 `PoseMetrics` 中新增 `issue_details` 字段，保存每个问题的原始得分
2. 在 `Supervisor.on_posture_update()` 中选择得分最高的问题
3. 告警消息只显示最严重的一个问题，例如："低头 (中度)"

### 2.4 缺少诊断日志用于回溯计算过程

问题：
1. 如果对某一次告警有异议，无法查看当时的算法计算细节
2. 缺少逐帧的原始数据记录

处理：
1. 新增 `src/diagnostic_log.py` - `DiagnosticLogger` 类
2. 记录每帧的完整计算结果（检测结果、姿势 metrics、距离数据、监督器状态）
3. 记录告警事件（包含照片路径）
4. 记录学习会话状态变化
5. 自动清理：最多保留 3 天数据
6. 提供查询接口 `query_logs()` / `query_alerts()`

### 2.5 姿势问题标签在生成时是英文

问题：
1. 问题标签在 `pose_detector.py` 中是英文生成
2. 然后在 `supervision.py` 中再翻译成中文
3. 这增加了复杂度且容易出错

处理：
1. 直接在 `pose_detector.py` 中用中文生成问题标签
2. 移除翻译逻辑，简化代码

## 3. 诊断日志使用说明

### 3.1 日志存储位置

默认在 `data/diagnostic_log.db`

### 3.2 查询示例

```python
from diagnostic_log import DiagnosticLogger

logger = DiagnosticLogger("data/diagnostic_log.db")

# 查询最近的告警
alerts = logger.query_alerts(limit=10)

# 查询某段时间的帧日志
logs = logger.query_logs(start_time=time.time() - 3600, limit=100)
```

### 3.3 数据结构

- `diagnostic_logs` 表：逐帧数据
- `alert_events` 表：告警事件（含照片路径）
- `session_events` 表：会话变化

## 4. 前版关键问题与处理

### 2.1 重启计数累计导致长期运行脆弱

问题：

1. 临时故障会永久占用重启次数

处理：

1. 主进程新增稳定运行后自动清零逻辑

### 2.2 推理端阻塞接收导致渲染卡顿

问题：

1. 旧版 `recv_frame()` 阻塞主循环
2. 影响渲染、温控和退出响应

处理：

1. 改为接收线程 + 主循环解耦

### 2.3 距离估算建立在固定方框上

问题：

1. 旧版用鼻子中心加固定方框估算距离
2. 不随真实远近可靠变化

处理：

1. 改为基于头部关键点估计头部区域
2. 保留轻量测距方案，避免引入重型模型

### 2.4 一套姿态规则硬适配所有机位

问题：

1. 正面与侧面监督重点不同
2. 一套规则容易误报

处理：

1. 新增 `front` / `side` 机位模式
2. 正面保留距离、低头、歪头
3. 侧面聚焦头前伸和趴桌

### 2.5 距离低置信度时累计被直接打断

问题：

1. 一两帧边缘抖动会导致距离累计清零

处理：

1. 增加低置信度宽限时间

### 2.6 会话数据完全丢失

问题：

1. 旧版只保留当前 session

处理：

1. 增加内存历史
2. 引入 SQLite 最小持久化

### 2.7 持久化重复写入与 socket 清理边界

问题：

1. 重复退出可能导致同一会话多次写库
2. receiver 重连与退出阶段的 socket 生命周期需要更清晰
3. 丢帧统计可能在两处重复累计

处理：

1. SQLite 增加唯一索引和 upsert
2. socket 关闭后立即置空，并统一由锁保护
3. 丢帧统计只保留接收线程一处累计

## 3. 当前技术取舍

### 3.1 为什么没有直接换成更重的检测算法

原因：

1. 当前目标硬件是树莓派 5 单机部署
2. 本轮优先级是稳定、可信、低误报
3. 先收敛问题边界，再考虑后续升级

### 3.2 为什么保留 MediaPipe Pose 为主

原因：

1. 当前已有成熟集成
2. 对上半身监督仍然够用
3. 与树莓派 5 的实时性目标基本匹配

### 3.3 为什么侧面机位不再强调距离

原因：

1. 侧面头部尺度受朝向影响太大
2. 更适合用来监督前倾和趴桌

---

## 5. 当前已知边界

1. 光照差时关键点稳定性仍会下降
2. 轻量距离估算仍存在个体差异，需要校准
3. 极端遮挡或极端角度下，结果仍可能波动
4. 诊断日志保留 3 天，占用空间随帧率增长
